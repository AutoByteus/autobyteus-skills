# 常见设计实践

本文件收集阶段 3（设计）和阶段 5（未来态运行时调用栈审查）中的实用共享设计实践。这些是可选实践，而非强制规则。当它们有助于主干、所有权或非主干关注点的清晰度时使用。

## 主干清单

### 什么是主干清单

- 设计范围内每条相关数据流主干的显式列表。
- 每条主干有一个 ID、范围（`主要端到端` / `返回-事件` / `有界局部`）和治理所有者。

### 何时使用

- 当范围涉及多个业务路径或有意义的返回/事件/有界局部流程时。
- 在涉及多个业务入口点的 `Medium` 和 `Large` 范围中。

### 实际指南

- 主要主干按用例/业务路径列出。允许并预期多条主要主干。
- 返回-事件主干在回调/响应/事件传播路径架构上有意义时列出。
- 有界局部主干在一个所有者内部的循环/状态机/调度器实质影响行为时列出。

## 权威边界规则

### 什么是权威边界规则

- 调用者使用一个领域主题的权威公共边界，而非同时使用该边界和其内部管理器/仓库/辅助工具/低级关注点之一。
- 这是 `无边界绕过 / 无混合层级依赖` 规则。

### 何时使用

- 当所有者有一个公共服务/存储/管理器层，且调用者可能被诱使直接到达低级内部时。
- 当审查可能暴露调用者同时依赖外层所有者和其内部之一时。

### 实际指南

- 如果调用者需要的功能内部拥有者有但外层边界未暴露，扩展边界 API 而非绕过边界。
- 审查意图：选择每个主题的一个权威边界供上层调用者使用；如果调用者需要内部，则边界有误或 API 不完整。

## 结构触发器

### 何时提取可复用的自有结构

- 当相同的数据结构、类型、规范化器、转换器、映射器或模式跨多个文件重复出现时。
- 提取到正确子系统所有权下的一个共享文件中。
- 保持共享核心紧凑。如果一个情况需要额外字段，使用有意义的专门化变体或组合而非扩展共享基础为万能形状。

### 何时分裂vs保持扁平

- 当扁平布局隐藏真实的所有权或结构深度时分裂。
- 当过度分裂创建人工的、无所有者的目录或可选模块分组时保持扁平。
- 使用判断：目标是可读的所有权和结构深度，而非每个主干步骤一个目录的机械规则。

### 何时移除/退役

- 当更清晰的子系统所有权、可复用的自有结构或更紧凑的文件职责使碎片化或重复的部分不再必要时。
- 将移除视为一等设计工作：在范围内命名并移除/退役它们。

## 重复协调所有权

### 问题

- 当协调策略（提供者选择/回退/重试/聚合/路由/扇出）跨调用者重复出现时，没有明确的所有者。

### 实践

- 为共享策略赋予一个明确的所有者，而非在多个调用者中重复它。

## 空间接间控制

### 问题

- 引入一个新的边界/层/包装器，它仅是透传的，不拥有任何策略、翻译或边界关注点。

### 实践

- 如果新的间接层不拥有真实的关注点，不要引入它。
- 审查应否决仅透传的边界。

## 共享结构 / 数据模型紧凑性

### 问题

- 共享基础类型变成了一个万能模型，大部分字段可选或在消费者间语义不同。
- 并行的重叠表示为相同概念共存。

### 实践

- 保持共享核心紧凑：每个字段一个明确含义。
- 当情况真正不同时使用有意义的专门化/组合，而非扩展共享基础。
- 将冗余属性从共享结构中移除。
- 合并或消除共享范围内的重叠并行表示。

## 命名-职责对齐

### 问题

- 范围漂移后名称不再匹配实际拥有的职责。

### 实践

- 当原始名称隐藏当前行为时重命名。
- 不要保留一个使特定于其名称的关注点变得不可见的名称。

## 现有结构偏见检查

### 问题

- 一个文件留在错误位置仅因为许多调用者已经从那里导入它。

### 实践

- 不要仅因现有结构偏见就保留错位的文件。
- 如果架构优先替代方案更清晰，移动文件并修复导入。

## 反 Hack 检查

### 问题

- 一个功能上有效的局部修复降低了数据流主干、所有权边界或非主干关注点。

### 实践

- 选择适当的结构变更以保持架构完整性；不要仅因局部 hack 更小就偏好它。
