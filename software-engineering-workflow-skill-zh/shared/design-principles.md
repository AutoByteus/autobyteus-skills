# 设计原则

本文件定义在阶段 3（设计）、阶段 5（未来态运行时调用栈审查）和阶段 8（代码审查）中使用的共享设计原则。

## 数据流主干清单和清晰度

- 每个有意义的行为路径都有一条数据流主干。
- 主干从发起表面延伸到有意义的下游后果。
- 主干清单列出给定范围内的所有相关主干。
- 主干节点是主要领域主题节点（处理、拥有或推进主干上数据/状态的实体）。
- 非主干关注点围绕主干描述，它们服务但不取代主干。
- 设计以主干清单为先进行组织；所有权和文件映射从主干推导。

## 主干跨度充分性规则

- 主要主干必须延伸足够远以暴露真实的业务路径，而不是仅覆盖局部编辑段落。
- 实际默认为 `4-5` 个有意义的节点，除非完整路径确实更小。
- 好的形状：
  - `浏览器 UI -> 会话引导 -> 运行时调用 -> 曝光组合器 -> 浏览器表面`
- 差的形状：
  - `曝光组合器 -> 浏览器表面`
- 如果较短的局部路径隐藏了谁发起了行为、哪个边界是权威的或什么下游效果重要，则主要主干太短。

## 所有权清晰度和边界封装

- 每个主干节点由一个明确的所有者拥有。
- 所有者拥有其主题的决策权。
- 所有权边界是所有者的公共合约。
- 调用者与边界交互，而非与所有者的内部交互。
- 依赖方向遵循所有权。

## 权威边界规则

- 每个领域主题有一个权威的公共边界。
- 边界上方的调用者必须使用该边界而非其内部拥有的机制。
- 这是 `无边界绕过 / 无混合层级依赖` 规则。
- 好的形状：
  - `调用者 -> ArtifactStore`
  - `ArtifactStore -> ArtifactRepository`
- 差的形状：
  - `调用者 -> ArtifactStore`
  - `调用者 -> ArtifactRepository`
  - `ArtifactStore -> ArtifactRepository`
- 如果调用者同时需要服务和其内部仓库，则边界有误或 API 不完整。
- 解决方式是扩展或重新设计边界，而非标准化绕过。

## 围绕主干的非主干关注点

- 非主干关注点服务主干上的所有者但不处于主干上。
- 它们有明确的服务所有者。
- 在推导非主干需求时，应首先复用或扩展现有能力域/子系统。
- 非主干关注点不是主干节点，不应接管主干逻辑。

## 现有能力复用

- 在创建新的临时辅助工具之前，检查现有子系统或能力域是否已经自然拥有该关注点。
- 当现有能力域适合时复用或扩展，而非为流程旁边的临时辅助创建新文件。

## 可复用的自有结构

- 当重复的数据结构、类型、规范化器、转换器、映射器或模式跨多个文件出现时，提取到正确子系统下的可复用自有文件中。
- 保持提取的共享结构紧凑：每个字段一个明确含义，移除冗余属性，在情况真正不同时使用有意义的专门化而非扩展共享基础为万能形状。

## 文件职责和放置

- 文件是关注点的主要具体映射目标。
- 每个文件拥有一个连贯的关注点。
- 文件放置遵循所有权：文件位于拥有其关注点的子系统和文件夹内。
- 可选的模块分组仅在有助于可读性时使用。
- 子系统是更广泛的所有权上下文；模块不是文件的同义词。

## 接口边界清晰度

- 每个接口/API/查询/命令/服务方法拥有一个主题和一个职责。
- 标识形状是显式的。
- 不要使用一个通用接口来混合不同的主题或标识形状。

## 范围适当的关注点分离

- 关注点分离应遵循所有权和非主干关注点，而非先于它们。
- 在 UI/前端范围中：职责在视图/组件/存储边界上清晰。
- 在非 UI 范围中：职责在文件和服务边界上清晰，子系统分组保持可读。
- 在集成/基础设施范围中：每个适配器或集成聚焦的文件/组拥有一个具有明确合约的集成关注点。

## 命名质量

- 文件名、文件夹名、API 名、类型/模式名、函数/方法名、参数名和局部变量名必须与其真实职责和行为匹配。
- 名称保持具体且不令人意外；拒绝误导性缩写、模糊占位符或隐藏副作用或所有权的名称。
- `主干`、`所有者` 和 `非主干关注点` 是关系术语，而非命名模板；拒绝当它们隐藏实际处理的具体关注点时使用 `Support`、`Supporting`、`OffSpine`、`SideConcern` 或 `Helper` 等通用名称。
