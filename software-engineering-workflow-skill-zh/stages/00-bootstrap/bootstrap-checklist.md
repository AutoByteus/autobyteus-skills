# 阶段 0 引导清单

在每个工单开始调查之前使用此清单。

## 必需顺序

1. 创建或复用 `tickets/in-progress/<ticket-name>/`。
2. 如果项目是 git 仓库，解析引导基础分支：
   - 当用户提供时使用显式指定的基础分支，
   - 否则推断置信度最高的跟踪远程默认/集成分支。
3. 如果项目是 git 仓库且需要新建工单工作树/分支，先刷新跟踪的远程引用。
4. 如果项目是 git 仓库，为 `codex/<ticket-name>` 创建或复用专用工单工作树；创建新工单分支时，从已解析基础分支的最新跟踪远程状态创建。
5. 从 `shared/workflow-state-template.md` 创建或更新 `tickets/in-progress/<ticket-name>/workflow-state.md`，并填写 `阶段 0 引导记录`。
6. 设置 `Current Stage = 0` 和 `Code Edit Permission = Locked`。
7. 创建或更新 `tickets/in-progress/<ticket-name>/requirements.md`，状态设为 `Draft`。

## 引导规则

- 在引导序列物理写入之前不要开始深入调查。
- 当现有工单文件夹和分支/工作树已匹配时复用它们。
- 如果用户指定了基础分支，始终使用该分支的最新跟踪远程状态，而非过时的本地分支。
- 不要从过时的本地基础创建新的工单分支/工作树；先刷新跟踪的远程引用。
- 先从用户输入或缺陷证据捕获草稿需求；不要等到完全理解后再行动。
- 如果远程刷新、基础分支解析或工单工作树创建失败，保持阶段 0 阻塞并记录阻塞项。
- 如果仓库不在 git 管理下，仍按相同顺序创建工单文件夹和阶段产物。

## 最低阶段 0 证据

- 工单路径存在
- git/非 git 引导决策已记录
- 若为 git 仓库：基础远程/基础分支已记录
- 若为 git 仓库：工作树路径 + 工单分支已记录
- 若为 git 仓库且创建了新工单工作树/分支：远程刷新结果已记录
- `workflow-state.md` 已初始化
- `requirements.md` 存在且状态为 `Draft`
