# 阶段 1 调查指南

调查是为了建立准确理解和验证假设所需的任何活动。

## 调查可以包括

将以下内容视为示例，而非封闭列表。选择手头问题所需的任何证据收集工作。

- 阅读代码、文档、配置、模式、日志和数据
- 阅读公共 API 文档、SDK 文档、规范、RFC、发布说明、问题追踪器和迁移指南
- 运行命令、复现、探针或追踪
- 编写小型脚本、聚焦测试或临时仪表来暴露当前行为
- 设置复现或缩小问题所需的最小化本地、容器化、模拟或基于固件的环境
- 当仅靠本地代码不足以解释行为时，检查上游、供应商、开源或合作伙伴源代码
- 当这是验证集成或复现失败的实际方式时，下载或克隆外部仓库、示例或示例应用
- 在需要时检查 API、外部参考或互联网资源

## 产物标准

- 原地更新 `tickets/in-progress/<ticket-name>/investigation-notes.md`。
- 将 `investigation-notes.md` 视为持久的调查档案，而非简短回顾。
- 写入足够详细的内容，使后续阶段可以直接复用发现，而无需重复相同的搜索/研究工作（除非事实已改变）。
- 使用简短的综合部分提高可读性，但在产物中保留具体的证据路径、URL、查询、命令、发现和影响。

## 必需内容

- 调查目标、问题或假设
- 范围分类（`Small` / `Medium` / `Large`）及理由
- 咨询的确切来源：
  - 本地文件路径
  - URL / 外部文档 / 公共 API / 规范 / 问题追踪器
  - 使用时的上游、供应商或示例仓库
  - 运行的命令
  - 对复现或隔离产生实质影响的设置步骤
  - 有实质意义的搜索查询
- 当前入口点、执行边界、所有者、模块、文件夹和可能的文件放置问题
- 使用复现、追踪、脚本、聚焦测试或设置工作时的运行时/探针发现
- 对理解产生实质影响的外部依赖、上游源代码或示例应用发现
- 约束、未知项和设计影响
- 足够的代码库/运行时/API 细节，使后续阶段不需要从头重新发现相同事实

## 调查规则

- 不要仅将结果保存在记忆中。
- 如果后续阶段需要该细节，不要将详细证据压缩为简短摘要。
- 使用互联网或外部文档研究时，记录确切来源和从中获取的具体事实或约束。
- 使用公共 API、上游代码、问题追踪器、发布说明或克隆/下载的示例仓库时，记录确切来源、版本/标签/提交（如可用）以及从中学到的具体事实或行为。
- 使用代码库搜索时，记录重要的确切文件、符号或路径，而不仅仅是结论。
- 使用命令、探针、追踪或复现时，记录运行了什么以及重要的结果。
- 当复现或隔离问题需要最小化环境设置、固件、模拟、特性标志或临时仪表时，将其视为有效的阶段 1 调查工作，并记录对结果产生实质影响的设置步骤。
- 小型诊断脚本、聚焦复现测试、临时日志和窄范围探针在其目的是证据收集而非交付修复时仍属于阶段 1 调查工作。
- 一次性调查沙盒、克隆的仓库、示例应用、临时测试工具和本地镜像在用于收集证据而非引入永久依赖或交付修复时是有效的阶段 1 工具。
- 如果测试、工具或代码更改旨在作为产品行为或持久验证套件的一部分保留，请在后续阶段而非阶段 1 中处理该后续工作。
- 如果后续阶段暴露了不清楚或跨领域的问题，重新打开阶段 1 并追加新证据，而非无声地替换早期发现。
- 调查应足够减少不确定性，使 `requirements.md` 能达到 `Design-ready`。
