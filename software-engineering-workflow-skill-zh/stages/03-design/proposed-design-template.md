## 术语

- `子系统` / `能力域`：拥有更广泛工作类别的较大功能区域，可能包含多个文件加可选的模块分组。
- `模块`：当代码库需要时，子系统内部的可选中间分组。在本模板中，`模块`不是一个文件的同义词，也不是默认的所有权术语。
- `文件夹` / `目录`：用于组织文件和任何可选模块分组的物理分组。
- `文件`：一个具体的源文件，也是一个具体关注点应落入的主要单元。

## 设计阅读顺序

按从抽象到具体的顺序阅读和编写此设计：

1. 数据流主干
2. 子系统 / 能力域分配
3. 草拟文件职责 -> 提取可复用的自有结构 -> 最终确定文件职责
4. 文件夹/路径映射

## 阅读规则

- 本文件必须首先围绕数据流主干清单进行组织。
- 主要领域主题节点和所有权边界是主要的设计叙事。
- 非主干关注点围绕其服务的主干进行描述。
- 当现有能力域/子系统自然适合非主干需求时，应复用或扩展它们。
- 文件是关注点的主要具体映射目标，子系统是更广泛的所有权上下文。可选的模块分组仅在有助于可读性时作为派生实现映射出现。
- 子系统、文件夹和文件映射应以主干和所有权为导向，但不应机械化。可选的模块分组仅在对读者有帮助时作为次级结构使用。

## 数据流主干清单

列出每条与理解设计相关的重要主干。

| 主干 ID | 范围（`主要端到端`/`返回-事件`/`有界局部`） | 起点 | 终点 | 治理节点 / 治理所有者 | 为何重要 |

规则：
- `主要端到端` 表示一个范围内用例或主要业务路径的主要顶层业务主干。当多条业务路径重要时，允许且预期有多行 `主要端到端`。
- `返回-事件` 表示对行为有意义的返回路径、回调路径或事件传播路径。它可能跨边界向外/向上/向回移动。
- 如果一个循环、worker 周期、状态机或调度器在一个所有者内部实质性地影响行为，为其添加一条 `有界局部` 主干，而非将其隐含。
- `有界局部` 表示一个所有者内部的内部流程。明确命名父所有者；此行添加局部细节，不取代更长的主要主干。
- `主干跨度充分性规则`（强制）：主要主干必须延伸足够远以暴露真实的业务路径，而非仅覆盖局部编辑段落。实际默认为 `4-5` 个有意义的节点，除非完整路径确实更小。

## 主要执行 / 数据流主干

将每条主要端到端线写成简短的箭头链。
本节可以有多条主要线。

示例：
- `前端 -> API -> 服务 -> 仓储 -> 数据库`
- `输入 -> RunManager -> Run -> RunBackend -> 运行时 -> 提供者`

硬性规则：
- 即使代码变更仅触及 `1-3` 个节点，主要主干仍必须延伸足够远以暴露发起表面、权威所有者边界和有意义的下游后果。
- 当需要更长的主要主干来正确判断业务含义时，有界局部主干不能取代它。

每条主要主干的必需产物形状：
- 主干 ID
- 简短箭头链
- 简短叙述
- 主要领域主题节点
- 治理所有者
- 为何跨度足够长

主干跨度示例：
- 好的形状：
  - `浏览器 UI -> 会话引导 -> 运行时调用 -> 曝光组合器 -> 浏览器表面`
- 差的形状：
  - `曝光组合器 -> 浏览器表面`

失败条件：
- 如果较短的局部路径隐藏了谁发起了行为、哪个边界是权威的或什么下游效果重要，则主要主干太短。

## 范围适当的关注点分离检查

- UI/前端范围：职责在视图/组件/存储边界上清晰。
- 非 UI 范围：职责在文件和服务边界上清晰，子系统分组仍可读。
- 集成/基础设施范围：每个适配器或集成聚焦的文件/组拥有一个具有明确合约的集成关注点。
- 所有权说明：关注点分离应遵循所有权和非主干关注点，而非先于它们。
- 文件放置说明：文件夹/路径应使所属关注点显而易见；平台特定文件不应位于无关或模糊的位置。
- 布局说明：当紧凑结构保持更清晰时可以接受，但过度扁平或过度分裂的布局应被纠正。
