# 未来态运行时调用栈审查

使用此文档作为未来态运行时调用栈质量和用例完整性的实现前门控。
此审查验证与目标（`to-be`）设计行为的一致性，而非与当前（`as-is`）代码的对等性。
为工单保持一个规范的 `future-state-runtime-call-stack-review.md` 路径。在同一文件中记录后续轮次，并将最新轮次视为权威，同时保留早期轮次作为历史。

## 审查意图（强制）

- 主要检查：未来态运行时调用栈是否为连贯且可实现的未来态模型？
- 非通过标准：与当前代码调用路径精确匹配。
- 共享原则规则：审查使用与阶段 3 相同的设计原则（`数据流主干清单和清晰度`、`所有权清晰度和边界封装`、`围绕主干的非主干关注点`、所有权驱动的依赖验证和 `权威边界规则`）。
- 现有能力规则：当应复用或扩展现有子系统或能力域时，审查必须否决临时的非主干关注点增长。
- 主干清单规则：审查必须验证所有相关主干在设计基础中被显式列出，包括当循环、worker 周期、状态机或调度器实质影响行为时的有界局部主干。
- 主要范围说明：`主要端到端` 是按用例/业务路径的，不是全局单一的。当业务范围确实包含多条顶层路径时，审查应接受多条主要主干。
- `主干跨度充分性规则`（强制）：审查必须否决停留在局部编辑段落且未延伸足够远以暴露真实业务路径、权威所有者边界和下游后果的主要主干。实际默认为 `4-5` 个有意义的节点，除非完整路径确实更小。
- 好的形状示例：
  - `浏览器 UI -> 会话引导 -> 运行时调用 -> 曝光组合器 -> 浏览器表面`
- 差的形状示例：
  - `曝光组合器 -> 浏览器表面`
- `权威边界规则`（强制）：审查必须否决调用者同时依赖权威外层边界及该边界内部低级关注点之一的设计，而应使用一个权威入口点。这是 `无边界绕过 / 无混合层级依赖` 规则。
- 好的形状示例：
  - `API 处理器 -> ArtifactStore -> ArtifactRepository`
- 差的形状示例：
  - `API 处理器 -> ArtifactStore`
  - `API 处理器 -> ArtifactRepository`
  - `ArtifactStore -> ArtifactRepository`
- 非必需操作：默认添加/移除层；仅在对此范围确实增加清晰度时描述分层。
- 示例清晰度规则：如果设计使用示例因为形状不显而易见，审查这些示例是否确实澄清了目标形状。
- 重复协调触发规则：如果协调策略跨调用者重复（提供者选择/回退/重试/聚合/路由/扇出），审查应要求更清晰的所有者。
- 空间接间规则：否决仅透传且不拥有任何策略、翻译或边界关注点的新边界。
- 局部修复不够规则：如果修复功能上有效但降低了主干、所有权或非主干关注点，标记 `Fail` 并通过分类重入要求架构产物更新。
- 任何需要设计/调用栈更新的发现都是阻塞性的。
- 无向后兼容审查规则：如果未来态行为为旧流程保留兼容性包装器、双路径逻辑或遗留回退分支，标记 `Fail`。
- 所有权依赖审查规则：如果未来态行为引入紧耦合、双向依赖纠缠或不合理的循环，标记 `Fail`。
