# 实现

使用此单一产物同时用于：
- 稳定的阶段 6 实现基线
- 实时的阶段 6 执行/进度记录
- 阶段 7、8 和 9 的简要下游移交/状态指针

写入：
- `tickets/in-progress/<ticket-name>/implementation.md`

文档纪律：
- `计划基线` 章节是预期的实现形状，仅在重新规划或分类重入需要时变更。
- `执行追踪` 章节是实时日志，应在实现期间和下游阶段状态实质变更时持续更新。
- `实现工作表` 是主要文件/任务追踪器。在其他地方优先使用变更 ID，而非重复相同的文件列表（除非确切路径是该章节的重点）。
- 在实现期间持续应用共享设计原则和常见设计实践。如果文件级现实揭示了更紧凑或更安全的形状，不要将审查过的设计产物视为机械完成。
- 详细的阶段 7、阶段 8 和阶段 9 记录属于它们各自的规范产物。在此仅保留移交输入和简短状态指针。
- 如果阶段 6 在 `workflow-state.md` 中记录了分类重入，默认立即执行第一个返回阶段，无需等待另一条用户消息。不要在仅记录路径和描述下一步后停止。

## 范围分类

- 分类：`Small` / `Medium` / `Large`
- 理由：
- 工作流深度：
  - `Small` -> 草拟 `implementation.md` 解决方案概要 -> 未来态运行时调用栈 -> 未来态运行时调用栈审查（迭代深度轮次直到 `Go Confirmed`）-> 最终确定 `implementation.md` 基线 -> 实现执行
  - `Medium` -> 设计提案文档 -> 未来态运行时调用栈 -> 未来态运行时调用栈审查（迭代深度轮次直到 `Go Confirmed`）-> `implementation.md` 基线 -> 实现执行
  - `Large` -> 设计提案文档 -> 未来态运行时调用栈 -> 未来态运行时调用栈审查（迭代深度轮次直到 `Go Confirmed`）-> `implementation.md` 基线 -> 实现执行

### 原则

- 自底向上：先实现依赖项，再实现依赖者。
- 测试驱动：在实现过程中同步编写单元测试和集成测试。
- 主干引导的实现规则：按主干和所有者排序工作；文件顺序从该结构派生，任何可选的模块分组遵循所有权模型而非引导它。
- 强制现代化规则：无向后兼容垫片或遗留分支。
- 强制清理规则：在阶段 6 关闭前移除范围内的死代码、过时文件、未使用的辅助工具/测试/标志/适配器和休眠的已替换路径。
- 强制所有权/解耦/关注点分离规则：保持清晰的子系统边界、单向依赖方向和范围适当的文件职责；避免添加紧耦合/循环或混合关注点文件。
- 强制 `权威边界规则`：当一个边界是某领域主题的预期公共权威时，不要让其上方的调用者同时依赖该边界及其内部的管理器、仓储、辅助工具或低级关注点之一。这是 `无边界绕过 / 无混合层级依赖` 规则。如果边界缺少所需的 API，扩展或重新设计它而非使绕过合理化。
- 强制 `主干跨度充分性规则`：实现必须保持更长的主要主干足够可见以保持业务路径的全局视图。当隐藏发起表面、权威所有者边界或下游后果时，不要将实现叙事压缩到仅局部编辑段落。实际默认为 `4-5` 个有意义的节点，除非真实路径确实更小。
- 多主要主干说明：如果范围包含多条顶层业务路径，实现可追溯性应保持所有相关主要主干而非强制将所有内容压入一个局部路径叙事。
- 强制共享结构连贯性规则：实现共享数据结构时，保持共享核心紧凑。如果一个情况需要额外字段，使用有意义的专门化变体或组合而非扩展共享基础为万能形状。
- 强制文件放置规则：将每个已触及文件保持在拥有其关注点的文件夹/边界中；当前放置误导时规划显式移动。
- 强制共享原则实现规则：在编码期间独立应用共享设计原则和常见设计实践。如果文件级实现细节揭示审查过的设计不完整、薄弱或错误，记录问题并分类为 `Design Impact` 而非在本地打补丁绕过。
- 强制主动大小压力规则：将阶段 8 源文件大小门控作为变更的源实现文件的活跃阶段 6 护栏。不要故意将变更的源实现文件扩展或留在 `500` 有效非空行以上。如果一个变更的源实现文件趋向该限制或在当前 diff 中超过 `220` 变更行，在实现期间拆分/重构/升级而非等到阶段 8。测试文件不使用此硬性源文件限制。
- 选择正确的结构变更以保持架构完整性；不要仅因局部 hack 更小就偏好它。
- 默认一次一个文件；仅在依赖边需要时使用有限的并行工作。
