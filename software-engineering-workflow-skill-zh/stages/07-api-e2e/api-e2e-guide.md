# 阶段 7 API/E2E 和可执行验证指南

## 目的

阶段 7 是实现和代码审查之间的可执行验证门控。
它证明已实现的系统跨系统的真实可执行边界满足验收标准和重要的设计风险行为，不仅通过 HTTP API 或仅浏览器流程。

## 输入

- 当前 `requirements.md`
- 当前设计基础
- 阶段 6 的当前实现输出

## 场景来源

- `Requirement`：验证显式需求行为
- `Design-Risk`：验证设计引入的技术或架构风险

## 强制验证结构

- 从 `requirements.md` 构建验收标准矩阵
- 将每个范围内验收标准映射到至少一个可执行验证场景
- 将每个相关主干映射到至少一个可执行场景，或标记为 `N/A` 并附理由
- 将 `相关主干` 明确视为：
  - 每条范围内的主要业务主干，
  - 每条有意义的返回/事件主干，
  - 每条对行为产生实质影响的有界局部主干
- 如果多条主要业务路径在范围内，明确映射每条主要主干而非将它们压缩为一个通用场景
- 当有界局部主干在范围内时，在验证推理中保持它附属于其父所有者；它增加局部证明但不取代更长的主要主干覆盖
- 在 `api-e2e-testing.md` 中记录场景来源、验证模式、平台/运行时目标、预期结果、命令或工具和结果
- 跨重跑保持一个规范的 `api-e2e-testing.md` 路径；默认不创建版本化副本
- 在轮次 `>1` 时，先重新检查之前未解决的失败并更新先前失败解决部分，然后再声明新的门控结果
- 将最新记录的验证轮次视为权威，同时保留早期轮次作为历史
- 跨重跑对相同场景复用相同的场景 ID；仅为新发现的覆盖创建新 ID
- 阶段 7 执行期间，`workflow-state.md` 应显示 `Current Stage = 7` 和 `Code Edit Permission = Unlocked`
- 将场景类型视为依赖问题而非硬编码：API、浏览器 UI、原生桌面 UI、CLI、进程/生命周期、集成和分布式系统检查在它们是被证明的真实边界时都是有效的阶段 7 形式

## 验证资产规则

- 首先从持久验证开始：添加或更新应保留在仓库中并约束未来变更的测试、工具和验证资产。
- 如果持久的仓库驻留验证不足以证明行为，继续使用更广泛的可执行验证，如临时脚本、探针、环境设置、容器化设置、模拟或仿真的依赖、CLI 工具、原生桌面自动化、生命周期/更新/重启检查、多进程或多节点编排，或浏览器/计算机自动化。
- 明确记录哪些验证资产是持久的仓库内新增，哪些是临时的仅验证方法或脚手架。
- 在结果记录后移除临时的仅验证脚手架，除非保留它明确有助于作为持久的未来覆盖。

## 场景模式示例

- 后端/服务 API：请求 -> 服务/领域执行 -> 持久化或下游效果 -> 返回的合约和错误行为。
- 浏览器 UI：用户操作 -> 渲染状态转换 -> 网络副作用 -> 最终成功或失败展示。
- 原生桌面更新器：版本 `from` -> 更新下载/应用 -> 重启/重新启动 -> 版本 `to` -> 持久化状态或迁移验证。
- Worker/进程生命周期：入队或触发 -> 后台循环/进程交接 -> 副作用 -> 完成、确认或持久状态转换。
- 分布式或多节点：节点 A 上的操作 -> 同步/复制/故障转移 -> 节点 B 上的可观察状态或行为。

## 可行性规则

- 如果场景在当前环境中不可行，记录确切原因、约束、补偿证据和残余风险。
- 如果场景对平台或生命周期敏感，记录相关的 OS/运行时/版本/包或更新源的具体信息，而非将阻塞减少为通用环境说明。
- 保持阶段 7 `Blocked` 直到用户明确豁免不可行的验收标准。

## 失败分类

- `Local Fix`：重跑 `Stage 6 -> Stage 7`
- `Design Impact`：重跑 `Stage 1 -> Stage 3 -> Stage 4 -> Stage 5 -> Stage 6 -> Stage 7`
- `Requirement Gap`：重跑 `Stage 2 -> Stage 3 -> Stage 4 -> Stage 5 -> Stage 6 -> Stage 7`
- `Unclear`：重跑 `Stage 0 -> Stage 1 -> Stage 2 -> Stage 3 -> Stage 4 -> Stage 5 -> Stage 6 -> Stage 7`
- 在 `workflow-state.md` 中记录重入路径后，默认立即恢复第一个返回阶段，无需等待另一条用户消息。不要在仅描述重跑路径后停止。

不要在修复仅通过降低所有权、主干清晰度或文件放置来工作时将其分类为 `Local Fix`。

## 退出门控

此阶段仅在所有可执行的范围内验收标准为 `Passed`、所有相关可执行的主要/返回-事件/有界局部主干有证据（或明确的 `N/A` 理由），且跨系统真实范围内边界无未解决的失败或阻塞时才完成。
