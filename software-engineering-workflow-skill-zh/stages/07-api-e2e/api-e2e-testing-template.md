# 阶段 7 可执行验证（API/E2E）

使用此文档进行阶段 7 可执行验证实现和执行。
阶段 7 可覆盖 API、浏览器/UI、原生桌面/UI、CLI、进程/生命周期、集成或其他可执行场景——当这些是被证明的真实边界时。
不要在此文件追踪单元/集成测试；那属于 `implementation.md`。
阶段 7 在阶段 6 实现（源代码 + 单元/集成）完成后开始。

验证理念：
- 首先持久化应留在仓库中并约束未来变更的可执行验证资产
- 然后在需要时使用更广泛的可执行验证来证明当前环境中的行为
- 仅在临时的仅验证脚手架对未来覆盖明确有用时保留它
- 为工单保持一个规范的 `api-e2e-testing.md` 文件；在同一文件中记录后续轮次，将最新轮次视为权威同时保留早期轮次作为历史

- 当前验证轮次：`1` / `2` / `3` / ...
- 触发阶段：`6` / `重入`

- 最新权威轮次：`1` / `2` / `3` / ...

轮次规则：
- 默认不创建版本化的阶段 7 文件。
- 在轮次 `>1` 时，先检查先前未解决的失败，然后再声明轮次完成。
- 将覆盖矩阵和场景状态更新到最新真相；将失败历史保留在下面的失败日志和轮次表中。
- 跨重跑对相同场景复用相同的 `场景 ID`。仅为新发现的覆盖添加新 `场景 ID`。

## 覆盖规则

- 每个关键需求必须映射到至少一个场景。
- 每个范围内验收标准（`AC-*`）必须映射到至少一个可执行场景。
- 每个范围内用例必须映射到至少一个场景，或明确标记为 `N/A` 并附理由。
- 已批准设计基础中的每条相关主干必须映射到至少一个场景，或明确标记为 `N/A` 并附理由。
- `Design-Risk` 场景必须包含明确的技术目标/风险和预期结果。
- 使用 `AV-` 前缀的稳定场景 ID（例如：`AV-001`）。
- 非结构化的仅手动测试不属于默认工作流。如果 OS/平台约束强制需要人工辅助执行步骤，记录确切步骤、捕获的自动化证据、约束和残余风险；不要将临时手动确认视为充分的阶段 7 覆盖。
- 当任何验收标准为 `Unmapped`、`Not Run`、`Failed` 或 `Blocked` 时（除非因不可行情况被用户明确标记为 `Waived`），阶段 7 不能关闭。
- 阶段 7 执行期间，`workflow-state.md` 应显示 `Current Stage = 7` 和 `Code Edit Permission = Unlocked`。
- 阶段 7 包含测试文件/工具实现和测试执行。
- 当持久验证属于代码库时，应首先添加或更新。
- 当需要证明持久仓库驻留测试单独无法证明的行为时，允许使用临时可执行验证。

## 场景形状示例（适配而非照搬）

- API/服务合约：创建或更新持久的 API 测试，证明请求、响应、验证和下游副作用。
- 浏览器 UI 流程：使用仓库驻留的浏览器测试或聚焦自动化来证明可见状态变化和面向用户的结果。
- 原生桌面更新器流程：证明版本 `from` -> 更新应用 -> 重新启动/重启 -> 版本 `to` 和持久化状态或迁移结果。
- Worker/进程流程：启动相关进程或工具，证明入队/触发 -> 后台执行 -> 可观察的完成状态。
- 多节点或分布式流程：搭建所需的本地或容器化拓扑，证明复制、故障转移或同步行为。

规则：
- 在轮次 `>1` 时，声明新的门控结果前先更新此轮次的先前失败解决检查。
- 跨轮次保持场景 ID 稳定，使先前失败解决和覆盖历史保持可追溯。
- 最终分类前运行调查筛查：
  - 如果问题范围跨领域、根因不清楚或置信度低，设置 `需要调查 = Yes` 并先更新 `investigation-notes.md`。
  - 如果问题明确有界且置信度高，可直接进行分类。
- 对每个失败场景，在重入前后更新验收标准矩阵状态。
- `Local Fix` 要求先更新产物，然后修复，然后重跑 `阶段 6 -> 阶段 7` 后再重跑受影响场景。
- `Design Impact` 要求 `需要调查 = Yes` 和设计产物更新前的调查检查点。
- 如果潜在修复引入兼容性包装器/双路径行为/遗留保留、降低解耦（紧耦合/循环）或将代码保留在错误的关注点文件夹中，分类为 `Design Impact`（而非 `Local Fix`）。
- 如果在设计影响调查期间发现需求级差距，重新分类为 `Requirement Gap`。
- 在必需的上游产物更新前不允许直接源代码修补。
- `Design Impact` 要求全链重入：`阶段 1 -> 阶段 3 -> 阶段 4 -> 阶段 5 -> 阶段 6 -> 阶段 7` 后再重跑受影响场景。
- `Requirement Gap` 要求全链重入：`阶段 2 -> 阶段 3 -> 阶段 4 -> 阶段 5 -> 阶段 6 -> 阶段 7` 后再重跑受影响场景。
- `Unclear`/跨领域根因要求全链重入：`阶段 0 -> 阶段 1 -> 阶段 2 -> 阶段 3 -> 阶段 4 -> 阶段 5 -> 阶段 6 -> 阶段 7`。
- 重入路径中的阶段 0 意味着在同一工单/工作树中重新打开引导控制（`workflow-state.md`、锁定状态、产物基线）；不创建新的工单文件夹。
