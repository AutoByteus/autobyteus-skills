# 阶段 8 代码审查指南

## 目的

阶段 8 是工程独立门控。
它在阶段 7 可执行验证门控通过后运行。
阶段 8 针对共享设计原则、常见设计实践和硬性审查门控检查变更后的代码库。

## 进入此阶段条件

- 阶段 7 门控为 `Pass`
- `workflow-state.md` 已移至阶段 8 并锁定源代码编辑

## 输入

- 当前 `workflow-state.md`
- 当前设计基础/实现产物
- 阶段 6 输出（源代码 + 单元/集成）
- 阶段 7 `api-e2e-testing.md`

## 规范产物

- 创建或更新 `tickets/in-progress/<ticket-name>/code-review.md`
- 使用 `code-review-template.md` 作为起始结构

## 审查必须覆盖

- 数据流主干清单和清晰度（包括主干跨度充分性和主干清单完整性）
- 所有权清晰度和边界封装（包括权威边界规则）
- API / 接口 / 查询 / 命令清晰度
- 关注点分离和文件放置
- 共享结构 / 数据模型紧凑性和可复用的自有结构
- 命名质量和局部可读性
- 验证强度
- 边缘情况下的运行时正确性
- 无向后兼容 / 无遗留保留
- 清理完整性

## 硬性文件大小策略

- 测量命令：`rg -n "\\S" <file-path> | wc -l`
- `>500` 有效非空行的任何变更源文件 => `Design Impact` + `Fail`
- 无软性 `501-700` 中间区间
- `>220` 变更行增量需要在 `code-review.md` 中记录设计影响评估
- 测试文件不使用此源文件大小限制

## 审查评分卡要求

- 总体 `/10` 和 `/100`
- 按以下规范顺序的十个分类行：
  1. 数据流主干清单和清晰度
  2. 所有权清晰度和边界封装
  3. API / 接口 / 查询 / 命令清晰度
  4. 关注点分离和文件放置
  5. 共享结构 / 数据模型紧凑性和可复用的自有结构
  6. 命名质量和局部可读性
  7. 验证强度
  8. 边缘情况下的运行时正确性
  9. 无向后兼容 / 无遗留保留
  10. 清理完整性
- 每行必须记录：分数、获得该分数的原因、薄弱之处、应改进之处
- 顺序是审查推理顺序，不是等权重列表
- 任何分类低于 `9.0` 都不视为干净通过
- 总体摘要永远不能覆盖阻塞项或失败的强制检查

## 审查门控规则

- 代码审查门控决策为 `Pass` 或 `Fail`
- 必须记录所有硬性审查门控检查
- 如果数据流主干清单和清晰度、所有权或安全性检查失败，门控为 `Fail` 无论其他分数如何
- 如果任何评分卡分类低于 `9.0`，门控为 `Fail`
- 如果任何变更源文件超过 `500` 有效非空行，门控为 `Fail`（`Design Impact`）

## 失败分类和重入

- `Local Fix`：`阶段 6 -> 阶段 7 -> 阶段 8`
- `Validation Gap`：`阶段 7 -> 阶段 8`（当主要问题是不充分的阶段 7 覆盖/证据而非代码或设计漂移时）
- `Design Impact`：`阶段 1 -> 阶段 3 -> 阶段 4 -> 阶段 5 -> 阶段 6 -> 阶段 7 -> 阶段 8`
- `Requirement Gap`：`阶段 2 -> 阶段 3 -> 阶段 4 -> 阶段 5 -> 阶段 6 -> 阶段 7 -> 阶段 8`
- `Unclear`/跨领域根因：`阶段 0 -> 阶段 1 -> 阶段 2 -> 阶段 3 -> 阶段 4 -> 阶段 5 -> 阶段 6 -> 阶段 7 -> 阶段 8`

在 `workflow-state.md` 中记录重入路径后，默认立即恢复第一个返回阶段，无需等待另一条用户消息。不要在仅描述重跑路径后停止。

不要在修复仅通过降低所有权、主干清晰度或文件放置来工作时将其分类为 `Local Fix`。

## 退出门控

此阶段仅在审查决策为 `Pass`、所有强制检查均满足、详细的优先级排序审查评分卡已记录、无评分卡分类低于 `9.0` 时才完成。
