# 代码审查原则

## 审查理念

- 代码审查不是风格检查。它是独立的工程门控。
- 审查根据共享设计原则和常见设计实践评估交付的代码。
- 审查使用详细的优先级排序评分卡使质量判断透明。
- 分类的优先级顺序是审查推理顺序，不是等权重清单。
- `数据流主干清单和清晰度` 是最高优先级，因为它驱动所有后续的所有权/结构/接口/放置推理。
- 最后两个分类（`无向后兼容 / 无遗留保留` 和 `清理完整性`）是硬性门控：如果它们失败，审查失败——无论其他分数如何。

## 硬性审查门控

以下检查是强制的且不可协商。任何一项失败都会导致整体审查 `Fail`，无论评分卡中的其他分数如何。

### 1. 源文件大小门控

- 测量：`rg -n "\\S" <file-path> | wc -l`
- 任何变更的源文件 `>500` 有效非空行 => `Design Impact` + `Fail`
- 无软性 `501-700` 中间区间
- `>220` 变更行增量需要在 `code-review.md` 中记录设计影响评估
- 测试文件豁免于此源文件大小限制

### 2. 无向后兼容 / 无遗留保留

- 如果变更保留向后兼容包装器、双路径行为或遗留回退分支：`Fail`
- 如果范围内的旧行为路径保留而非移除：`Fail`

### 3. 清理完整性

- 如果范围内可识别的死代码、过时文件、未使用的辅助工具/测试/标志/适配器或休眠的已替换路径仍然存在：`Fail`

### 4. 权威边界规则

- 如果调用者同时依赖权威外层边界及其内部拥有的低级关注点之一：`Fail`

### 5. 所有权驱动的依赖方向

- 如果引入了新的不合理紧耦合或循环依赖：`Fail`

### 6. 主干跨度充分性

- 如果主要实现主干压缩到仅局部编辑段落，隐藏了发起表面、权威所有者边界或下游后果：`Fail`

### 7. 文件放置

- 如果已触及文件位于错误的子系统或目录下：`Fail`（除非阶段 6 记录了该放置为有意的且有设计理由）

### 8. 共享结构连贯性

- 如果共享基础类型扩展为万能模型或保留了冗余/重叠的并行表示：`Design Impact` 或降低评分

## 评分卡分类（规范顺序）

优先级排序反映审查推理流程：更高优先级的分类在设计意义上优先，且在当前轮次失败时阻止有意义地评估后续分类。

| 优先级 | 分类 | 审查焦点 |
| --- | --- | --- |
| 1 | 数据流主干清单和清晰度 | 相关主干是否被列出？主要主干跨度是否足够？主干清单是否完整？流程是否可按主干和所有者阅读？ |
| 2 | 所有权清晰度和边界封装 | 每个主干节点是否有一个明确的所有者？边界是否封装内部？权威边界规则是否得到保持？ |
| 3 | API / 接口 / 查询 / 命令清晰度 | 每个公共边界是否拥有一个主题和一个职责？标识形状是否显式？ |
| 4 | 关注点分离和文件放置 | 文件是否位于正确的子系统和目录中？可选模块分组是否有理由支持？关注点分离是否遵循所有权？ |
| 5 | 共享结构 / 数据模型紧凑性和可复用的自有结构 | 共享基础是否紧凑？是否已提取可复用自有结构？是否移除了冗余属性？ |
| 6 | 命名质量和局部可读性 | 名称是否与真实职责匹配？是否有误导或模糊的名称？ |
| 7 | 验证强度 | 单元/集成/可执行验证覆盖是否充分？关键路径和边缘情况是否有证据？ |
| 8 | 边缘情况下的运行时正确性 | 在错误条件、竞争状态、空/nil 情况和边界条件下行为是否正确？ |
| 9 | 无向后兼容 / 无遗留保留 | 范围内是否有遗留代码路径或兼容机制残留？ |
| 10 | 清理完整性 | 范围内的死代码、过时文件、未使用的辅助工具/标志/适配器和休眠路径是否已移除？ |

## 评分卡规则

- 每个分类必须有一个数字分数加简短叙述
- 叙述必须包含：分数、获得该分数的原因、薄弱之处、应改进之处
- 任何分类低于 `9.0` 都不是干净通过
- 总体摘要（`/10` 和 `/100`）用于趋势可见性
- 总体摘要永远不能覆盖失败的分类或阻塞的硬性门控检查

## 通用审查原则

- 优先选择保持架构完整性的结构变更而非较小的局部 hack
- 不要仅因现有结构偏见就保留错位的文件
- 如果修复功能有效但降低了主干/所有权/非主干关注点，仍然无效
- 审查时检查补丁叠补丁模式：如果后续修复在非自然所有者中积累且引入紧耦合，在分类中反映
- 重复协调策略应有明确的所有者而非跨调用者分散
