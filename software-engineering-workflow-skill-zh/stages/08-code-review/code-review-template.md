# 代码审查

使用此文档进行阶段 7 可执行验证通过后的 `阶段 8` 代码审查。
此门控强制执行结构质量、源文件可维护性和强制重入规则。

## 源文件大小和结构审计（强制）

此审计仅适用于变更的源实现文件。
测试文件仍在审查范围内，但不受 `>500` 硬限制或 `>220` 变更行增量门控约束。
仅将调查笔记和早期设计产物用作上下文。如果它们与共享原则、实际代码或清晰的审查发现冲突，适当分类问题而非遵从早期产物。

| 源文件 | 有效非空行数 | 添加/扩展功能（`Yes`/`No`） | `>500` 硬限制检查 | `>220` 变更行增量门控 | 范围适当的关注点分离检查（`Pass`/`Fail`） | 文件放置检查（`Pass`/`Fail`） | 初步分类（`N/A`/`Local Fix`/`Validation Gap`/`Design Impact`/`Requirement Gap`/`Unclear`） | 必需操作（`保持`/`拆分`/`移动`/`重构`） |
| --- | --- | --- | --- | --- | --- | --- | --- | --- |
|  |  |  | Pass/Fail/N/A | Pass/Fail/N/A | Pass/Fail | Pass/Fail |  |  |

规则：
- 对每个变更源文件使用显式测量命令：
  - 有效非空行数：`rg -n "\\S" <file-path> | wc -l`
  - 变更行增量：`git diff --numstat <base-ref>...HEAD -- <file-path>`
- 不要将测试文件放在此表中；在代码审查的其他位置审查其清晰度、可维护性和证据质量。
- 强制基线使用有效非空行数。
- 对于有效非空行数 `<=500`，正常审查适用。
- 硬限制规则：如果任何变更源文件的有效非空行数 `>500`，默认分类为 `Design Impact` 且阶段 8 决策为 `Fail`。
- 对于 `>500` 硬限制情况，默认不继续；先应用重入映射。
- 此工作流中无软性中间区间（`501-700`）且无默认例外路径。
- 增量门控：如果单个变更源文件在当前 diff 中有 `>220` 变更行，即使文件大小 `<=500` 也记录设计影响评估。
- 关注点分离规则：如果变更文件混合了不相关的职责或在一个边界中隐藏了多个所有者，标记 `范围适当的关注点分离检查 = Fail` 并要求 `拆分`/`重构`。
- 文件放置规则：如果文件路径/文件夹遮蔽了所属关注点或将平台特定代码放在不相关的位置，标记 `文件放置检查 = Fail` 并记录必需的 `移动`/`拆分` 操作。
- 阶段 8 期间，`workflow-state.md` 应显示 `Current Stage = 8` 和 `Code Edit Permission = Locked`。

## 结构完整性检查（强制）

将 `权威边界规则` 视为此表中最高信号的结构检查之一。
将 `主干跨度充分性规则` 也视为硬性检查：主要审查主干必须足够长以暴露真实业务路径而非仅局部编辑段落。

快速示例：
- 好的主干跨度：
  - `浏览器 UI -> 会话引导 -> 运行时调用 -> 曝光组合器 -> 浏览器表面`
- 差的主干跨度：
  - `曝光组合器 -> 浏览器表面`
- 好的形状：
  - `调用者 -> 服务`
  - `服务 -> 仓储`
- 差的形状：
  - `调用者 -> 服务`
  - `调用者 -> 仓储`
  - `服务 -> 仓储`
- 审查解读：
  - 如果调用者需要同时使用 `服务` 和 `仓储`，要么服务不是真正的权威，要么调用者在绕过权威
  - 不要用模糊的依赖措辞隐藏这一点；明确记录为权威边界失败

## 审查评分卡（强制）

在每个审查轮次记录评分卡，包括失败轮次。
评分卡解释当前质量；它不覆盖阶段 8 门控。
使用以下规范分类顺序。顺序反映审查推理路径而非等权重分类列表。

- 总体评分（`/10`）：
- 总体评分（`/100`）：
- 评分计算说明：报告 `/10` 和 `/100` 仅用于摘要/趋势可见性。如果报告总体评分，下面十个分类的简单平均可以接受，但平均值永远不是阶段 8 的通过/失败规则。

| 优先级 | 分类 | 分数（`1.0-10.0`） | 为何此分数 | 薄弱之处 / 拖累因素 | 应改进之处 |
| --- | --- | --- | --- | --- | --- |
| `1` | `数据流主干清单和清晰度` |  |  |  |  |
| `2` | `所有权清晰度和边界封装` |  |  |  |  |
| `3` | `API / 接口 / 查询 / 命令清晰度` |  |  |  |  |
| `4` | `关注点分离和文件放置` |  |  |  |  |
| `5` | `共享结构 / 数据模型紧凑性和可复用的自有结构` |  |  |  |  |
| `6` | `命名质量和局部可读性` |  |  |  |  |
| `7` | `验证强度` |  |  |  |  |
| `8` | `边缘情况下的运行时正确性` |  |  |  |  |
| `9` | `无向后兼容 / 无遗留保留` |  |  |  |  |
| `10` | `清理完整性` |  |  |  |  |

规则：

- 不要仅留下原始数字；每行必须解释分数、薄弱之处和预期改进。
- 遵循列出的优先级顺序。评分卡不是无序检查清单。
- 分类 `1` 必须包含主干跨度判断。如果任何相关主要审查主干不够长以暴露真实业务路径，分类 `1` 不能干净通过。
- 每个分类都是强制的。干净的阶段 8 通过目标是每个分类 `>= 9.0`。任何分类低于 `9.0` 都是真实差距，通常应导致审查失败。
- 不要让总体摘要覆盖薄弱分类。门控仍遵循强制检查和阻塞性发现。
- 如果一个分类没有被大量行使，仍对相关变更边界的质量评分并在理由列中解释有限范围。

## 重入声明（`Fail` 时强制）

- 触发阶段：
- 分类（`Local Fix`/`Validation Gap`/`Design Impact`/`Requirement Gap`/`Unclear`）：
- 必需的返回路径：
  - `Local Fix` -> 先更新实现产物 -> 代码修复 -> 重跑 `阶段 6 -> 阶段 7 -> 阶段 8`
  - `Validation Gap` -> 更新阶段 7 验证覆盖/证据并重跑 `阶段 7 -> 阶段 8`
  - `Design Impact` -> `阶段 1 -> 阶段 3 -> 阶段 4 -> 阶段 5 -> 阶段 6 -> 阶段 7 -> 阶段 8`
  - `Requirement Gap` -> `阶段 2 -> 阶段 3 -> 阶段 4 -> 阶段 5 -> 阶段 6 -> 阶段 7 -> 阶段 8`
  - `Unclear`/跨领域 -> `阶段 0 -> 阶段 1 -> 阶段 2 -> 阶段 3 -> 阶段 4 -> 阶段 5 -> 阶段 6 -> 阶段 7 -> 阶段 8`
  - 重入路径中的阶段 0 意味着在同一工单/工作树中重新打开引导控制（`workflow-state.md`、锁定状态、产物基线）；不创建新的工单文件夹。
