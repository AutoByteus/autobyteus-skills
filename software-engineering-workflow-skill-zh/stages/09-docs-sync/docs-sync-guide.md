# 阶段 9 文档同步指南

在阶段 8 完成后使用此指南。

## 为何此阶段存在

- 工单产物解释了一个变更是如何交付的，但它们不是持久的真相来源
- 长期的项目理解应在工单归档后继续存在
- 阶段 9 将稳定的设计、运行时、所有权和操作知识推广到长期文档中
- 此阶段很重要，因为未来的工程师应能从项目文档理解当前系统，而无需从旧工单重建历史

## 规范策略

- 长期项目真相属于项目文档，如 `docs/` 和 `ARCHITECTURE.md`
- 工单产物保持为任务本地记录，不是永久真相来源

## 阶段 9 产物

- 创建/更新 `tickets/in-progress/<ticket-name>/docs-sync.md`
- 用于记录：
  - 审查了哪些长期文档，
  - 更新了哪些文档，
  - 推广了哪些持久的设计/运行时知识，
  - 移除或替换了什么，
  - 或为什么确实无文档影响

## 变更时更新

- 子系统、文件、相关时的可选模块分组或公共接口
- 运行时流程或操作行为
- 重命名、移动或移除的组件
- 随功能变化的测试或操作程序

## 良好阶段 9 输出应解释什么

- 什么改变了
- 为什么改变
- 当前的架构/运行时形状现在是什么
- 哪些边界、子系统、文件或 API 现在需要理解
- 哪些旧组件、路径或概念被移除或替换
- 哪些操作或验证预期发生了变化

## 必需输出

- 当现有文档已覆盖该功能时原地更新
- 仅在当前无文档覆盖该功能时创建缺失文档
- 在 `docs-sync.md` 中记录 `Updated` 或 `No impact` 及理由
- 当一个规范文档可以原地更新时避免重复重叠文档

## 阶段 9 规则

- 不要仅因测试通过就跳过文档同步
- 如果无文档影响，明确说明
- 不要将此视为美化清理；文档必须解释代码库的当前状态
- 当工单本地的设计知识应成为持久项目知识时，将其推广到 `docs/` 或其他规范文档中，而非仅留在 `tickets/` 下
- 仅当文档与最终实现行为匹配时文档同步才算完成

## 阻塞 / 重入规则

- 如果因最终实现状态或工单产物仍需小的具体纠正而无法使文档如实编写，分类为 `Local Fix` 并重跑 `Stage 6 -> Stage 7 -> Stage 8 -> Stage 9`。
- 如果文档暴露了缺失或歧义的预期行为，分类为 `Requirement Gap` 并重跑 `Stage 2 -> Stage 3 -> Stage 4 -> Stage 5 -> Stage 6 -> Stage 7 -> Stage 8 -> Stage 9`。
- 如果最终实现状态或预期行为太不清楚或跨领域而无法如实记录，分类为 `Unclear` 并重跑 `Stage 0 -> Stage 1 -> Stage 2 -> Stage 3 -> Stage 4 -> Stage 5 -> Stage 6 -> Stage 7 -> Stage 8 -> Stage 9`。
- 如果文档仅因外部文档系统或文档访问阻塞而无法完成，保持阶段 9 `Blocked` 并停留在阶段 9 直到阻塞解除。
- 在 `workflow-state.md` 中记录重入路径后，默认立即恢复第一个返回阶段，无需等待另一条用户消息。
