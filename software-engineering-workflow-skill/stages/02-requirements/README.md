# Stage 2 Requirements

## Purpose

Turn rough requirement capture into a design-ready behavioral contract.
This stage makes intended behavior explicit enough to support design, runtime modeling, and later Stage 7 validation.

## Enter This Stage When

- Stage 1 investigation is current
- a later stage exposes a requirement gap or ambiguity

## Stage-Owned Outputs

- `tickets/in-progress/<ticket-name>/requirements.md`
- stable `requirement_id` and `acceptance_criteria_id` values
- requirement-to-use-case and acceptance-criteria-to-scenario coverage intent

## Exit Gate

Leave Stage 2 only when `requirements.md` is `Design-ready` or `Refined` and the expected behavior is clear enough to support Stage 3 design and later Stage 7 validation.

## Local Files

- `requirements-refinement-guide.md`: required quality, sections, mappings, and refinement rules
